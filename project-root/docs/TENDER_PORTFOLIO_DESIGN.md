# Портфель тендеров — Продуктовая спецификация

## 1. Контекст и задачи тендерного отдела

Типовой процесс tender management охватывает 8 функциональных зон:

1. **Поиск и мониторинг** — сбор тендеров из внешних источников (ЕИС, закупки.gov, коммерческие площадки), фильтрация по критериям.
2. **Квалификация** — Go/No-go решение на основе требований, рисков, маржинальности.
3. **Планирование подготовки** — таймлайн, задачи, ответственные, SLA до дедлайна.
4. **Документооборот** — ТЗ, формы, справки, шаблоны ответов, версионирование.
5. **Командная работа** — согласования с финансами, юристами, производством, пресейлом.
6. **Финмодель и ценообразование** — расчёт себестоимости, маржи, пределов для аукциона.
7. **Участие в торгах** — электронные торги, переторжка, контроль статусов на площадках.
8. **Аналитика** — воронка, win-rate, причины проигрыша, сегментация.

## 2. Информационная архитектура страницы «Портфель тендеров»

Страница строится по принципу **дашборд + операционная панель** — сверху аналитика и KPI, ниже инструменты работы.

| 6 | **Аналитика (мини-графики)** | Win-rate, динамика, сегментация |

## 3. Функциональные стеки (домены)

### 3.1 Возможности и воронка тендеров
**Назначение:** Управление жизненным циклом тендера от появления до закрытия.

**Основные экраны/виджеты:**
- **Pipeline-воронка** — горизонтальные колонки по стадиям, drag-and-drop перемещение.
- **Карточка тендера** — номер, название, заказчик, площадка, сумма, дедлайн, стадия, вероятность, ответственный.
- **Быстрые действия** — «Перейти к подготовке», «Открыть аукцион», «Дублировать».
- **Фильтры:** площадка, заказчик, дата, сумма, стадия, ответственный, тип (44-ФЗ/223-ФЗ/коммерческий).

**Стадии воронки (по умолчанию):**
1. Новый / На рассмотрении
2. Квалификация (Go/No-go)
3. Подготовка заявки
4. Согласование цены
5. Подача заявки
6. Аукцион / Переторжка
7. Ожидание результатов
8. Выигран / Проигран
9. Договор подписан

### 3.2 Планирование и задачи
**Назначение:** Трекинг подготовки заявки, контроль SLA.

**Основные экраны/виджеты:**
- **SLA-доска** — Kanban задач по тендеру: «Сбор документов», «Согласование ТЗ», «Финмодель», «Юридическая проверка».
- **Gantt-подготовки** — таймлайн от публикации до дедлайна с milestones.
- **Виджет «Критические дедлайны»** — тендеры с <48 часов до подачи/аукциона.

### 3.3 Документы и шаблоны
**Назначение:** Централизованное хранилище документов по тендеру.

**Основные экраны/виджеты:**
- **Панель документов тендера** — список файлов по категориям (ТЗ, формы, справки, договор).
- **Шаблоны** — быстрая вставка типовых блоков (описание компании, референсы, технические характеристики).
- **Версионирование** — история изменений документа, сравнение версий.

### 3.4 Команда и согласования
**Назначение:** Маршруты согласования, распределение ролей.

**Основные экраны/виджеты:**
- **Карточка команды тендера** — роли: лид тендера, пресейл, финансист, юрист, производство.
- **Лента согласований** — кто согласовал, кто отклонил, комментарии.
- **Уведомления** — push/email о необходимости согласования.

### 3.5 Финмодель и оценка
**Назначение:** Расчёт цены, маржинальности, стратегии аукциона.

**Основные экраны/виджеты:**
- **Калькулятор себестоимости** — прямые/косвенные затраты, накладные.
- **Маржинальность** — целевая vs фактическая, визуализация в карточке тендера (цветовой индикатор).
- **Критерии оценки** — таблица критериев заказчика, максимальные баллы, расчёт вероятного набора баллов.
- **Предел снижения** — нижняя граница цены для аукциона.

### 3.6 Аукционы и торги
**Назначение:** Отслеживание и участие в торгах.

**Основные экраны/виджеты:**
- **Панель активных аукционов** — таймер обратного отсчёта, текущая цена, шаг торгов, предел снижения.
- **История ставок** — таблица с историей изменения цены.
- **Стратегия** — автоматические/ручные ставки, правила снижения.

### 3.7 Аналитика и отчётность
**Назначение:** Измерение эффективности тендерного отдела.

**Основные экраны/виджеты:**
- **Воронка конверсии** — количество и сумма по стадиям.
- **Win-rate** — общий, по заказчикам, по площадкам, по типам тендеров.
- **Причины проигрыша** — pie-chart: цена, технические критерии, отсутствие референсов и т.д.
- **Динамика** — график поданных/выигранных тендеров по месяцам.
- **Топ заказчиков и конкурентов** — таблицы с метриками.

### 3.8 Справочники и мастер-данные
**Назначение:** Единые справочники для автозаполнения и фильтрации.

**Основные экраны/виджеты:**
- **Заказчики** — реестр заказчиков с историей.
- **Площадки** — настройки подключения к ЭТП (логины, сертификаты).
- **Конкуренты** — база конкурентов, история цен.
- **Типы тендеров** — 44-ФЗ, 223-ФЗ, коммерческие, международные.
- **Номенклатура** — типовые позиции для быстрого расчёта.

```
├─────────────────────────────────────────────────────────────────┤
│  ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐          │
│  │ Активные │ │ Объём    │ │ Win-rate │ │ На       │          │
│  │ 12       │ │ 45.2 млн │ │ 23%      │ │ аукционе │          │
│  │ ▲ +3     │ │ ▲ +8.1   │ │ ▼ -2%    │ │ 3        │          │
│  └──────────┘ └──────────┘ └──────────┘ └──────────┘          │
│  ┌──────────┐ ┌──────────┐                                     │
│  │ Просроч. │ │ Средняя  │                                     │
│  │ задачи 2 │ │ подготов.│                                     │
│  │ ▼ -1     │ │ 14 дн    │                                     │
│  └──────────┘ └──────────┘                                     │
└─────────────────────────────────────────────────────────────────┘
```

**KPI тайлы:**
- Активные тендеры (кол-во + динамика)
- Общий объём активных тендеров (сумма НМЦ / начальных цен)
- Win-rate (% выигранных от поданных за период)
- На аукционе (кол-во тендеров с активными торгами)
- Просроченные задачи (критично)
- Среднее время подготовки (дней от Go до подачи)

### 4.2 Воронка тендеров (Pipeline)

```
┌─────────────┬─────────────┬─────────────┬─────────────┬─────────────┐
│ НОВЫЙ       │ КВАЛИФИКАЦ.│ ПОДГОТОВКА │ СОГЛАСОВАН. │ ПОДАНО      │
│ 8 шт        │ 5 шт        │ 12 шт      │ 4 шт        │ 6 шт        │
│ 120 млн ₽   │ 45 млн ₽    │ 210 млн ₽  │ 89 млн ₽    │ 156 млн ₽   │
├─────────────┼─────────────┼─────────────┼─────────────┼─────────────┤
│ [Карточка]  │ [Карточка]  │ [Карточка] │ [Карточка]  │ [Карточка]  │
│ [Карточка]  │             │ [Карточка] │             │ [Карточка]  │
│             │             │ [Карточка] │             │             │
└─────────────┴─────────────┴─────────────┴─────────────┴─────────────┘
     ↓ 62%         ↓ 80%         ↓ 33%         ↓ 100%
```

**Карточка тендера в воронке:**
- Номер тендера (ссылка)
- Название закупки (1 строка)
- Заказчик (иконка + название)
- НМЦ / объём
- Дедлайн подачи / аукциона
- Прогресс-бар подготовки (если на стадии подготовки)
- Маржинальность (цветной индикатор: зелёный >30%, жёлтый 15-30%, красный <15%)
- Ответственный (аватар)
- Быстрые действия: «Открыть», «Задачи», «Документы»

### 6.2 Описание модулей

#### Tender Core
**Read:**
- `GET /tenders` — список с фильтрами, пагинацией, сортировкой
- `GET /tenders/{id}` — детальная карточка тендера
- `GET /tenders/pipeline` — агрегат по стадиям (count + sum)
- `GET /tenders/{id}/history` — история изменений стадий

**Write:**
- `POST /tenders` — создание тендера
- `PATCH /tenders/{id}/stage` — смена стадии
- `PATCH /tenders/{id}` — обновление полей
- `POST /tenders/{id}/clone` — дублирование

#### Task / Workflow
**Read:**
- `GET /tenders/{id}/tasks` — задачи по тендеру
- `GET /tasks/my` — мои задачи с фильтрами
- `GET /tasks/overdue` — просроченные

**Write:**
- `POST /tenders/{id}/tasks` — создание задачи
- `PATCH /tasks/{id}/status` — изменение статуса
- `POST /tasks/{id}/assign` — назначение исполнителя

#### Document Service
**Read:**
- `GET /tenders/{id}/documents` — список документов
- `GET /documents/templates` — список шаблонов
- `GET /documents/{id}/versions` — версии

**Write:**
- `POST /tenders/{id}/documents` — загрузка файла
- `POST /documents/{id}/generate` — генерация из шаблона
- `DELETE /documents/{id}` — удаление

#### Pricing & Scoring
**Read:**
- `GET /tenders/{id}/pricing` — финмодель
- `GET /tenders/{id}/scoring` — расчёт баллов

**Write:**
- `PUT /tenders/{id}/pricing` — сохранение финмодели
- `POST /tenders/{id}/pricing/calculate` — пересчёт маржи

#### Auction Connector
**Read:**
- `GET /auctions/active` — активные аукционы
- `GET /auctions/{id}/bids` — история ставок
- `GET /auctions/{id}/status` — статус с площадки

**Write:**
- `POST /auctions/{id}/bid` — подача ставки
- `POST /auctions/{id}/strategy` — настройка автостратегии

#### Analytics Service
**Read:**
- `GET /analytics/dashboard` — KPI для дашборда
- `GET /analytics/funnel` — воронка конверсии
- `GET /analytics/winrate` — win-rate с сегментацией
- `GET /analytics/loss-reasons` — причины проигрыша

**Write:**
- `POST /analytics/events` — запись событий (async)

#### Master Data
**Read:**
- `GET /customers` — заказчики
- `GET /platforms` — площадки
- `GET /competitors` — конкуренты

**Write:**
- CRUD для всех справочников

#### Notification Service
**Read:**
- `GET /notifications` — список уведомлений

**Write:**
- `POST /notifications/send` — отправка (async, вызывается другими сервисами)

## 7. MVP vs Расширение (приоритизация)

### MVP (Релиз 1) — Must Have
Для типичного тендерного отдела из 3-10 человек критичны:

| # | Функционал | Обоснование |
|---|-----------|-------------|
| 1 | **Реестр тендеров + воронка** | Основной рабочий инструмент, без него нет системы |
| 2 | **Карточка тендера с базовыми полями** | Номер, название, заказчик, сумма, дедлайн, стадия, ответственный |
| 3 | **Смена стадий и базовая история** | Отслеживание жизненного цикла |
| 4 | **Задачи по тендеру (todo-список)** | Контроль подготовки, дедлайны |
| 5 | **Загрузка документов** | Хранение ТЗ, форм, справок |
| 6 | **Базовая аналитика (KPI)** | Активные, win-rate, объём |
| 7 | **Справочники: заказчики и площадки** | Для фильтрации и автозаполнения |

## 8. Рекомендации по UI/UX

### Принципы дизайна страницы

1. **Двухуровневая навигация:** верхний уровень — стадии воронки (как табы), нижний — фильтры и реестр.
2. **Цветовая кодировка стадий:** каждая стадия имеет свой цвет, единый во всём интерфейсе.
3. **Акцент на срочность:** тендеры с дедлайном <48 часов подсвечиваются, аукционы — отдельным блоком.
4. **Единая карточка тендера:** одинаковый компонент в воронке, таблице и дашборде.
5. **Быстрые действия:** основные операции (смена стадии, назначение задачи) — не более 2 кликов.
6. **Мобильная адаптация:** реестр превращается в карточки, аукционы — пуш-уведомления.
